# 证券公司港美股经纪业务数据看板 - 后端架构设计

## 1. 整体架构设计

### 1.1 架构层次

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              客户端层                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                       dashboard-rest (REST接口服务)                       │
├─────────────────────────────────────────────────────────────────────────────┤
│             dashboard-bridge (桥接服务)        dashboard-srv (业务服务)   │
├─────────────────────────────────────────────────────────────────────────────┤
│                       dashboard-data (数据服务)                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                  数据库 / 缓存 / 文件系统 / SSO                           │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 服务间依赖关系

- **dashboard-rest** 依赖于 **dashboard-srv** 和 **dashboard-bridge**
- **dashboard-srv** 依赖于 **dashboard-data**
- **dashboard-bridge** 依赖于 **dashboard-data**
- **dashboard-data** 直接与底层基础设施交互

## 2. 服务详细设计

### 2.1 dashboard-rest (REST接口服务)

**职责**：
- 向前端提供 RESTful 风格接口
- 请求参数校验（基于 Schema 配置）
- 响应格式化
- 接口文档生成
- 测试数据 Mock

**技术栈**：
- Golang + Gin
- Apifox 文档
- Validator 校验
- Redis 缓存

**关键特性**：
- 轻量化，高并发
- 无业务逻辑
- 严格控制长连接和 IO
- 关键事件通知
- 支持接口测试

**目录结构**：
```
├── main.go              # 应用入口
├── internal/            # 内部包
│   ├── api/             # API 层，处理HTTP请求和响应
│   ├── grpc/            # gRPC 客户端，用于调用业务/桥接服务
│   │   ├── client/      # gRPC 客户端代码
│   │   └── proto/       # protobuf 定义文件
│   ├── validator/       # 请求参数校验
│   ├── mock/            # 测试数据Mock
│   └── model/           # 数据模型
├── pkg/                 # 公共包
│   ├── config/          # 配置
│   ├── logger/          # 日志
│   └── utils/           # 工具函数
├── configs/             # 配置文件
├── scripts/             # 脚本
└── Makefile             # 构建脚本
```

### 2.3 dashboard-bridge (桥接服务)

**职责**：
- 对接第三方服务（银行、柜台、咨询等）
- 处理第三方服务的异常和错误
- 并发控制
- 数据转换和适配

**技术栈**：
- Golang + Worker 池
- Circuit Breaker 模式
- Retry 机制
- 消息队列

**关键特性**：
- 处理第三方服务的不确定性
- 保证数据一致性
- 支持异步处理

**目录结构**：
```

├── internal/            # 内部包
│   ├── api/             # API 层
│   ├── adapter/         # 第三方服务适配器
│   ├── worker/          # 并发处理
│   ├── circuit/         # 熔断器
│   └── model/           # 数据模型
```

## 3. 技术规范

### 3.1 代码结构规范

- 严格遵循公司微服务架构规范中的目录结构
- 包名：小写，使用简短的单字或组合词
- 文件名：小写，使用下划线分隔
- 结构体：大驼峰命名法
- 函数：大驼峰命名法
- 变量：小驼峰命名法
- 常量：全大写，使用下划线分隔

### 3.2 接口规范

- RESTful 风格
- 使用 HTTP 标准方法（GET, POST,  DELETE）
- 状态码使用 HTTP 标准状态码
- 响应格式统一：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {}
  }
  ```

### 3.3 服务间通信规范

- **dashboard-rest 与 dashboard-srv/dashboard-bridge**：使用 gRPC 进行通信
- **dashboard-srv 与 dashboard-data**：使用 gRPC 进行通信
- **dashboard-bridge 与第三方服务**：根据第三方服务的接口类型选择合适的通信方式
- **异步通信**：使用消息队列（如 Kafka、RabbitMQ）进行异步处理

### 3.4 错误处理

- 统一错误码
- 错误信息国际化
- 详细的错误日志
- 适当的错误重试机制

### 3.5 日志规范

- 结构化日志（JSON 格式）
- 日志级别：DEBUG, INFO, WARN, ERROR, FATAL
- 包含必要的上下文信息
- 支持日志轮转

### 3.6 监控规范

- 服务健康检查
- 关键指标监控（QPS, 响应时间, 错误率）
- 分布式追踪
- 告警机制

## 4. 部署与运维

### 4.1 容器化

- **Docker 打包**：所有服务采用 Docker 方式打包
- **统一 Dockerfile**：为每个服务提供标准化的 Dockerfile
- **镜像版本管理**：使用语义化版本号管理 Docker 镜像

### 4.2 Kubernetes 部署

- **服务编排**：使用 Kubernetes 进行服务编排
- **配置管理**：使用 Kubernetes ConfigMap 和 Secret 管理配置信息和敏感数据
- **服务发现**：利用 Kubernetes Service 实现服务间的发现和通信
- **存储管理**：根据服务需求选择合适的存储方案
- **网络策略**：配置网络策略，控制服务间的网络访问

## 6. 总结

本后端架构设计严格遵循公司微服务架构规范，采用分层设计理念，将系统分为 REST 接口层、业务服务层、桥接服务层和数据服务层。各层职责明确，边界清晰，通过 gRPC 进行高效通信，确保系统的可维护性、可扩展性和可靠性。

在实施过程中，应根据实际业务需求和技术条件，对架构进行适当调整和优化，以达到最佳的系统性能和用户体验。
